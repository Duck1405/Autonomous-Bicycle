How to Write a Good Prompt
Foundational read for Project Brass 2.0. The single most important piece of training for trainers, get this right and the rest of the project gets easier.

Guide
Levels of prescriptiveness
Your job on Project Brass 2.0 is to write a matched pair of prompts, a goal prompt (just the goal) and a prescriptive prompt (full workflow), for the same sub-workflow inside an Apple app. The prescriptive prompt must be specific enough that an experienced user of the app could read it and produce one consistent output, every time. This page is the foundational read. Take 5 minutes with it before drafting your first pair.

How to write a good prompt

A strong Brass 2.0 prompt:

Starts with a realistic user need.
Names the app and the specific objects the user should act on.
Gives enough constraints to produce one consistent output.
Stays inside the assigned app and sub-workflow.
Avoids click-by-click UI choreography unless the exact menu choice matters.
Calls out any seeded state and before file requirements the workflow depends on.
The core idea
Write as if you're explaining a workflow step-by-step to a smart friend who's never used the app. Every meaningful step gets named, but you describe the action's purpose ("open settings"), not the literal UI gesture ("click the gear icon in the top-right corner").

The prescriptive prompt
Write the task as a detailed, unambiguous workflow. Describe actions and intent, not UI gestures like "click the button in the top right". This is the comparison baseline the model's plan will be graded against.

Key markers:

Starts with the app + action ("Open the Clock app. Navigate to Alarms…").
Lists concrete steps in order.
Uses specific names/values/dates/labels (no placeholders).
Has a clear, verifiable end state.
Targets the same end state as the goal prompt, same objects, same constraints, same result.
See levels of prescriptiveness →
The dual prompt approach
Every Brass 2.0 task ships two prompts for the same workflow. You write both, and they're submitted together as a matched pair. Together they show how much the model improves when it goes from a bare goal to a fully specified workflow, and where its confidence starts to break down.

Why two prompts
The goal prompt tests whether the model can plan from a goal alone. The prescriptive prompt establishes the ground-truth workflow.

The goal prompt
State only the goal, the way a real user would ask. No steps, no app navigation, no constraints beyond the outcome. The model is expected to produce its own plan from this prompt.

Key markers:

Starts with context and the user's "why" ("I'm getting ready for a wedding…", "Now that Valerie is out of school…").
Describes the desired end state in plain language.
Does not name menus, settings, or step order.
Sounds like something a person would actually say to an assistant.
The golden rule for both prompts
Concrete and specific always wins where it's allowed. The prescriptive prompt must leave no room to guess. The goal prompt must leave no room to misread the goal, vague-on-method is fine, vague-on-outcome is not.

Write in first person
Preferred style for both prompts in the pair. They should sound like the user speaking directly ("I need to clean up my watchlists…").

Before: too weak
The user needs to reorganize their Safari bookmarks.

After: meets the bar
I need to reorganize my Safari bookmarks before my presentation tomorrow.

What changed

First-person phrasing reads like a real request from a real person. Third-person framing distances the prompt from a believable user situation.

Common mistakes to avoid
Writing in third person or passive voice instead of first person.
Writing a tutorial instead of a request ("First you should go to…").
Describing UI gestures instead of actions ("click the button in the top right").
Adding hedged branches in the goal prompt that a real user wouldn't think about ("if notifications are on…"). The prescriptive prompt can and should include real conditional steps when the workflow requires them.
Using generic test data ("a contact named Person A").
Writing a single-step task ("open Mail and check my inbox").
What an acceptable seed description looks like
If your prompt assumes existing content, the seeded-state description should give a reviewer enough detail to reproduce the starting state without inventing anything. Name the objects, their counts, and any details the prompt actually references.

Before: too weak
There's a note about Q3 stuff.

After: meets the bar
Notes app contains one note titled "Q3 Planning" in the Work folder. The note has three bullet points: "Finalize budget," "Schedule kickoff," and "Draft team update." No attachments.

What changed

The strong version names the app, the exact title, folder, and contents the prompt acts on. The weak version forces a reviewer to guess.

How prescriptive is prescriptive enough?
Project Brass 2.0 needs very prescriptive prompts. Most trainers over-estimate how prescriptive their own prose is. The bar is higher than you think.

Two failure modes bracket the bar:

Too vague. "Manage your inbox." A reviewer can't tell what task that refers to, and multiple totally different outputs satisfy it.
Too literal. "Click pixel 247, 88." This is a UI script rather than a task instruction, and it breaks the second the UI changes.
Aim for the middle: every meaningful step is named, in order, by the result it produces. Not the gesture that produces it.

Three self-checks before you submit
Run your prompt through these three checks. If any one fails, edit before submitting.

Work through each of the following...

Checklist
0 of 4 complete


Could an experienced user of the app follow this and produce one consistent output?
If two reasonable people would produce different outputs from your prompt, it's not prescriptive enough.

Are all the explicit requirements ones that map to real workflow steps in the app?
If you've added requirements the app can't actually deliver (e.g., 'export as MP4 from Stickies'), the prompt isn't grounded.

Did I avoid being either too vague ("manage your inbox") or too literal ("click pixel 247, 88")?
Vague = scope is undefined. Literal = UI gestures, not task purpose. Aim for the middle.

Did I run through the steps myself in the app to make sure nothing is missing?
Walking the workflow yourself is the fastest way to catch a forgotten step or a wrong menu name.
Quick reference
Length. No fixed word count. Prescriptive enough = however many words it takes. Average ~3-8 sentences.
Tone. Realistic workplace request. Write it the way you'd actually ask a colleague.
Scope. Stay inside the sub-workflow you've been assigned (you'll see it in the Task Brief). If your prompt spills into other sub-workflows in the same app, that's fine, flag them in the Multi-step block. Out of scope: multi-app prompts, third-party integrations, deprecated features, per-app exclusions (see the Scope catalog).
No AI. Prompts are 100% human-written for Project Brass 2.0. If you draft with AI assistance, the screener catches it and LLM use is grounds for immediate offboarding.
Files. Some Brass 2.0 tasks require PII-free before files, or both. If your prompt assumes existing content, the seeded-state answer and before files must match it.

Before files
Use this page when a Brass 2.0 task needs files in addition to the prompt. The goal is simple: reviewers should be able to inspect what existed before the workflow started.

The rule
Before file

The PII-free starting artifact the prompt assumes already exists.

PII is the highest-risk miss
Every file should be synthetic, public-domain, or created only for the task. If a reviewer can identify a real person, place, account, workplace, or private detail, the task should be sent back.

Four-step workflow
1
Read the prompt
Ask: does this workflow need something to exist before the user starts?

2
Decide seeded state
If it edits, imports, annotates, updates, or organizes existing content, seeded state is usually Yes.

3
Prepare before files
Upload the PII-free artifact that represents the starting state when a file can capture it.

4
Attest in Handshake AI
After the Brass 2.0 upload form, return to the task and complete the file-upload attestation.

Which files usually matter?
App	Before
Contacts	Synthetic .vcf or .csv contact/card data
Calendar	Synthetic .ics event/calendar data
Preview	PII-free PDF, image, or form
Notes	Synthetic .txt, .rtf, .md, or zipped .rtfd notes
Photos	PII-free image/video set
Voice Memos	Written description of what is being said
QuickTime Player	PII-free media file
Stickies	Synthetic .txt, .rtf, or zipped .rtfd content
PII mini-check
No real names, emails, phone numbers, addresses, account numbers, IDs, payment details, or meeting links.
No identifiable faces, private interiors, license plates, inbox screenshots, or visible account UI.
No real contacts, calendar events, reminders, mail messages, notes, or workplace references from your own life.
Use fictional data, synthetic examples, or public-domain/CC0 source material.
Common file gotchas
Photos and HEIC metadata

Photos taken on a phone can include GPS/location metadata. This warning is here because PII can live in metadata even when the visible image looks safe. Use public-domain images when possible, or inspect and strip metadata before upload.

Live Photos

A Live Photo is a pair of files. If the prompt depends on Live Photo behavior, upload both the image and video components.

.rtfd bundles

.rtfd looks like a single file in macOS, but it is a folder bundle underneath. Compress it to .zip before upload.

Voice recordings

Keep recordings short and synthetic. Do not include real names, background voices, or personal/workplace details.

When files are ready
Upload every required before file, and supporting bundle through the Brass 2.0 file upload form. Then return to Handshake AI and complete the upload attestation before submitting.

Preview and Notes files
Preview and Notes usually need actual files because the prompt depends on a specific artifact or specific note text. Use this page as the practical recipe sheet after you understand the general before file rules.

Preview
Before artifact plus marked-up, filled, combined, or exported after artifact when required.

Seeded-state answer names the artifact and the final file expectation.

Notes
Before note files, plus after/exported note files when the task produces them.

Seeded-state answer names note count, folder/tag setup, and final-state difference.

Sourcing for Preview
Public-domain PDFs

Project Gutenberg, Library of Congress, public-domain government forms.

Public-domain images

Wikimedia Commons with CC0/Public Domain filter, or other clearly reusable image sources.

Trainer-created artifacts

A synthetic sketch, blank form, or non-personal handwritten scan created only for the task.

Sourcing for Notes
Synthetic notes

Best default. Write fictional project notes, lists, tags, checklists, or agendas from scratch.

Public-domain text

Short Project Gutenberg passages or public-domain material adapted into notes.

Structured examples

Synthetic tables, fictional directories, fake meeting agendas, or invented task lists.

PII check before upload
No real names, emails, phone numbers, addresses, IDs, financial details, or meeting links.
No faces or private images unless the source is public-domain and appropriate.
Check PDF metadata fields such as Author, Title, and Keywords.
Check image metadata for GPS/location before upload.
Replace your own city, employer, contacts, or life details with fictional equivalents.

Reviewer Rubric
How reviewers grade Project Brass 2.0 tasks. This is the grading instrument used by R1 reviewers, separate from the in-task rubric self-check.

Reviewer rubric
The Reviewer rubric below is for graders. R1 reviewers read your prompt, seeded-state answer, before files, and upload attestation against these criteria.

What reviewers grade
The reviewer rubric below is the current v1 cut. It will expand as patterns emerge and pilot data lands.

Prescriptiveness. Does every meaningful step appear in the prompt as an explicit instruction? Could an experienced user reproduce one consistent output from this alone? See How to write a good prompt for the bar, and acceptable levels of prescriptiveness for the range that passes review.
Scope fidelity. Does the prompt target the assigned sub-workflow per the Scope catalogue? Are per-app exclusions respected?
Realism. Does the prompt read like a real workplace request, or like a UI script? Is the level of detail what a colleague would actually write?
Grounding. Are the requirements ones the app can actually deliver? No invented features, no deprecated workflows.
Seeded-state accuracy. If the prompt assumes pre-existing content, is that state marked correctly and described clearly enough for a reviewer to recreate?
File completeness. When the workflow requires files, are the before file(s) present and correctly named?
Human authorship. Is the prompt 100% human-written? If the prompt looks AI-assisted, approve the task with the LLM-confirmed flag set, resolve the comment with a brief note (e.g., "confirmed"), select Approve, then add the task ID to MD's Role Change Form.

Frequently Missed Issues
The errors that most often cause Brass 2.0 tasks to be sent back, and how to avoid them before submitting.

Read this before your first submit
Most Brass 2.0 rework comes from four places: prompts that are not prescriptive enough, seeded-state answers that do not match the prompt, before files that are missing or mismatched, and accidental PII. Use this page as the common-error tab before you submit.

Reviewer decision rule
Reviewers separate issues into two severities:

Major - sent back for revision. One Major issue is enough to fail review.
Minor - approved with a coaching note. The reviewer resolves the issue, approves the task, and leaves a note so the fellow can fix the habit on the next task.
First task anxiety
Your first task is meant to catch misunderstandings early. If it is sent back, revise it carefully and ask for clarification in Slack. The goal is alignment before you scale volume, not a one-strike removal.
